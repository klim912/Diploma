Інструкція з генерації документації
Огляд
Документація проекту GameStore генерується автоматично з TSDoc-коментарів у коді за допомогою інструменту TypeDoc.

Передумови

Node.js >= 18
npm >= 9


Встановлення TypeDoc
bashnpm install --save-dev typedoc

Генерація документації
bashnpm run docs
Документація буде згенерована у папку docs/api/.
Відкрийте docs/api/index.html у браузері для перегляду.

Конфігурація
Налаштування TypeDoc знаходяться у файлі typedoc.json у корені проекту.

Стандарти документування (TSDoc)
Що обов'язково документувати

Всі публічні функції, хуки, класи, сервіси
Всі інтерфейси та типи (interface, type)
Складні алгоритми та неочевидні архітектурні рішення

Мінімальний шаблон для функції
typescript/**
 * Короткий опис (одне речення).
 *
 * Детальний опис (опціонально) — пояснення логіки, бізнес-правил.
 *
 * @param назва - Опис параметра
 * @returns Опис того, що повертається
 *
 * @example
 * ```ts
 * myFunction(arg); // очікуваний результат
 * ```
 */
Мінімальний шаблон для React-компонента
tsx/**
 * Короткий опис компонента.
 *
 * @remarks
 * Бізнес-логіка, взаємодія з іншими компонентами.
 *
 * @param props - Пропси компонента
 * @returns JSX-елемент
 *
 * @example
 * ```tsx
 * <MyComponent prop="value" onAction={handler} />
 * ```
 */
Обов'язкові теги
ТегДе використовувати@paramКожен параметр функції/методу@returnsЯкщо функція повертає значення@exampleБудь-яка публічна функція@remarksАрхітектурні рішення, бізнес-логіка@throwsЯкщо функція може кидати виняток

Правила для всіх, хто долучається до проекту

Кожна нова публічна функція/компонент — обов'язково з TSDoc-коментарем.
При зміні сигнатури функції — оновлюй коментар одночасно зі змінами коду.
Запускай npm run docs і перевіряй, що нові сутності з'являються в документації.
Не залишай @param без опису — це гірше, ніж відсутність документації.


CI/CD
Документація автоматично генерується та публікується на GitHub Pages при кожному push у гілку main.
Конфігурація: .github/workflows/docs.yml
